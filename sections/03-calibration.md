# III. Calibration

Calibration is matching stance to actual conditions. The stance described in Section I is right disposition. The Loop described in Section II is right method. Calibration is choosing which tools and how much rigor to apply to the work currently in front of the operator. Wrong stance applied to right conditions fails. Right stance applied to wrong conditions fails. Calibration is a prerequisite for the methodology to do useful work.

Two dimensions describe the conditions: the level of uncertainty, and the type of uncertainty. The first asks how much the operator already knows; the second asks whether the answer exists or is being constructed. The two together determine the appropriate stance.

## Levels of uncertainty

Four levels:

**Low.** The operator has done this exact kind of work before. The pattern is recognizable; the steps are familiar; the result will resemble prior results. Stance: execute precisely, prevention-weighted, no surprises expected.

**Medium.** The work resembles things the operator has done, but some elements are new. Stance: state assumptions, verify the key points before relying on them, expect that one or two assumptions will turn out wrong.

**High.** New territory. Multiple unknowns at once. Stance: research before building, propose the approach before executing, treat the first attempt as exploration, expect rework.

**Unknown.** The operator does not know what they do not know. Stance: stop. Ask. Do not guess. The cost of acting on Unknown as if it were High is severe; treat the boundary between High and Unknown as the most important calibration boundary in the document.

The levels are not absolute. They describe the operator's actual position relative to the work, not the work's intrinsic difficulty. A task that is High for one operator can be Low for another with relevant prior experience.

## Types of uncertainty

Two types:

**Search uncertainty.** The answer exists. Finding it is the work. Looking up how a function behaves, locating where a feature is implemented, identifying which library version handles a case correctly. All of these are search problems. The answer is somewhere; the question is where.

The stance for search is directed exploration. Form a hypothesis about where the answer lives, check it, refine. Verify systematically; once the answer is found, narrow to it. Search rewards persistence and method; it does not reward broad creative exploration.

**Generate uncertainty.** The answer is being constructed. Designing an architecture, drafting prose, choosing between multiple valid approaches, defining new terminology. All of these are generate problems. The answer does not exist yet; the operator is making it.

The stance for generate is open exploration. Produce multiple drafts. Take permission to be wrong on the first attempt. Broaden the option space before narrowing to a choice. Generate rewards range and willingness to discard; it does not reward locking on the first viable answer.

The two types call for opposite movements. Treating a search problem as generate produces wandering and waste. Treating a generate problem as search produces premature convergence on whatever the operator finds first.

## Matching stance to conditions

Stance comes from level and type together.

A Low-search problem is a quick lookup followed by verification. A Low-generate problem is producing a familiar pattern, with light review.

A Medium-search problem is a directed lookup with explicit verification of the assumptions being relied on. A Medium-generate problem is drafting against the known constraints, then checking that the draft holds against the unknowns.

A High-search problem is systematic exploration with documentation of what was checked and what remains uncertain. A High-generate problem is multiple drafts, deliberate divergence before convergence, and explicit permission to discard early attempts.

Unknown is the level at which the type is not yet identifiable. Before treating a problem as Search or Generate, the operator confirms that the type is identifiable. If it is not, the work is to discover what the problem is, not to attempt either approach.

The cost of miscalibration is paid in the work. Treating a High-generate problem as Low-generate produces an answer that arrives quickly and fails to reach the territory the problem opened. Treating a Low-search problem as High-generate produces wandering when a quick lookup would have settled the question. The type/level distinction matters more than it might seem: a one-step level error tends to be self-correcting, but treating search as generate (or vice versa) sends the work in the wrong direction.

## Calibration when stuck

When work is not progressing, when the operator has spent an unusual amount of time without an unusual amount of progress, the first check is calibration.

The questions to ask:

- What am I assuming about the level of uncertainty? What would change if my assumption were wrong by one level?
- What am I assuming about the type of uncertainty? Am I searching when I should be generating, or generating when I should be searching?
- What evidence would tell me my calibration is wrong?

If the answers to these questions reveal a mismatch, the recovery is to recalibrate and resume. The stance that fits the actual conditions begins working immediately; the stance that did not fit was producing the stuck state.

If the answers do not reveal a mismatch and the operator is still stuck, the problem is not stance. It is frame.

## Saturation and return

Frame-stuck is the state where multiple approaches keep failing the same way. The operator has tried what fits the apparent problem. The approaches do not work. The work is stuck not because of wrong stance, but because the framing of the problem is itself wrong.

The recovery is saturation and return. Leave the failing frame entirely. Move into a different domain: different code, different problem, different kind of work, different surface. Saturate there until the original frame has lost its grip. Return to the original problem with the framing space open again.

Saturation and return is not procrastination. It is the deliberate use of unrelated work to release the frame that has captured the operator's reasoning. The working hypothesis is that frames are sticky inside their context and release when the context changes substantially; the unrelated work is the deliberate context change.

The returning operator usually does not bring an answer. They bring a different question. The different question is one that did not surface inside the original frame. Whether the question leads to an answer is downstream; what saturation and return produces is the freedom to ask it.
